# DeepSeek Think 模式修复 — 完整记录

> 从报错到根因定位，到可维护、可适配多模式、零干预的最终方案。
> 最后一次更新：2026-07-16

---

## 0. 三代问题总览

本项目先后修复了 DMX 网关对 thinking 模式的三种**不同**校验规则触发的 400 错误：

| 代 | 错误信息 | 触发条件 | 章节 |
| --- | --- | --- | --- |
| 第一代 | `The content[].thinking in the thinking mode must be passed back to the API` | assistant 消息含 tool_use 但完全无 thinking 块 | 第 1-6 章 |
| 第二代 | `The reasoning_content in the thinking mode must be passed back to the API` | 请求**结尾**的 assistant 消息含未完成的 tool_use（CC 的 max_tokens 续写请求） | 第 10 章 |
| 第三代 | `unknown variant image_url, expected text` | 消息含 Anthropic 原生 `image` 内容块（截图 base64），DeepSeek 不支持多模态 | 第 11 章 |

三者触发条件各不相同，是 DMX 侧先后引入的三条独立校验规则。第三代在同一天（2026-07-16）紧接第二代之后发现，进一步说明 DMX 网关会持续变更校验逻辑且不同模型变体使用不同的校验路径，这类 shim 需要长期跟踪维护并充当"格式净化器"角色。

## 1. 问题（第一代）

Claude Code 通过 API 聚合商（DMX 等）调用 DeepSeek 等第三方模型时，thinking 模式下报错：

```text
API Error: 400 The content[].thinking in the thinking mode must be passed back to the API.
```

触发条件：EFFORT Extra high（启用 extended thinking）+ multi-turn + tool_use。

---

## 2. 根因

### 2.1 直接原因（两个机制叠加）

1. **CC 丢弃 thinking 块**：DeepSeek 经 DMX 后返回的 `thinking` 块 `signature` 是伪 UUID，不符合 Anthropic 签名规范，CC 在构建下一轮消息时丢弃它。
2. **上游 round-trip 校验**：DMX 要求如果 assistant 消息含 `tool_use`，则必须同时有 `thinking` 块。

两步叠加：CC 丢弃 thinking → 下一轮 assistant 只剩 `tool_use` → DMX 400。

### 2.2 关键实验

通过阶梯式 curl 探测确认根因：

| 实验 | assistant 消息内容 | 结果 |
| --- | --- | --- |
| B1 | `[thinking(UUID签名), tool_use]` | 200 |
| B2 | `[tool_use]`（无 thinking） | **400** — 精确复现 |
| B3 | `[thinking(空签名""), tool_use]` | 200 |
| B4 | `[thinking(签名"x"), tool_use]` | 200 |

**决定性证据**：上游只校验 thinking 块的存在性，不校验内容。

### 2.3 因果链

```
DeepSeek 无条件返回 thinking 块
  → CC 检测 signature 不是真 Anthropic 签名，丢弃 thinking 块
  → 下一轮 assistant 只剩 tool_use
  → DMX 400：content[].thinking must be passed back
```

---

## 3. 修复方案

### 3.1 核心思路

在 CC 和上游之间插入本地反向代理（shim），双路修复：

**请求侧**：检测到 deepseek 请求时，在 assistant 消息前注入占位 thinking 块。
```json
{ "type": "thinking", "thinking": "", "signature": "deepseek-think-fix" }
```

**响应侧**：在上游返回的 SSE/JSON 中，将非标准 signature 清空，让 CC 保留 thinking 块内容不丢弃。

### 3.2 当前架构

```
CC ──▶ shim :8788 (shim.js) ──▶ upstream (cc-switch 代理 或 dmxapi.cn 直连)
          │
          ├─ 请求侧处理（POST /v1/messages）：
          │    ├─ 读 settings.json 构建三类别名映射
          │    ├─ 查静态改写规则（SHIM_MODEL_REWRITE_RULES）
          │    ├─ 改写 body.model 从 CC 标签到真实模型名
          │    ├─ deepseek-* 且缺 thinking 块 → 注入占位块
          │    └─ 结尾 assistant 消息含 tool_use → 剥离未完成 tool_use（见第10章）
          ├─ 响应侧处理：
          │    ├─ 非流式 JSON：thinking signature 清零
          │    └─ SSE 流：逐 event 改写 thinking signature（content_block_start / signature_delta / delta）
          ├─ 每 3 秒轮询 settings.json：
          │    ├─ BASE_URL 自愈 + 别名映射刷新
          │    └─ trailing comma 容忍性解析
          ├─ 上游 4xx/5xx → 落盘完整请求/响应体到 error-dumps/（本地调试用）
          ├─ GET /health → 返回运行状态 JSON
          └─ 崩溃 → watchdog 3 秒后重启
```

### 3.3 模型识别（三类字段全覆盖）

shim 从 `settings.json` 中读取三类模型字段，构建别名映射：

| 字段类型 | shim 用途 |
| --- | --- |
| `ANTHROPIC_DEFAULT_<SLOT>_MODEL` / `_MODEL_NAME` | 映射 key → 真实模型名 |
| `ANTHROPIC_REASONING_MODEL` / `_MODEL_NAME` | 同上 |
| `ANTHROPIC_MODEL` / `_MODEL_NAME` | 同上 |

**解析优先级**：`SHIM_MODEL_REWRITE_RULES` > settings.json 别名映射 > identity（透传）

## 4. 进程管理与自愈

### 4.1 进程保护

| 层 | 机制 | 说明 |
| --- | --- | --- |
| L1 | watchdog 循环守护 | Node 崩溃 → 3 秒后自动重启 |
| L2 | uncaughtException / unhandledRejection | 致命异常记日志后 exit，触发 L1 |
| L3 | 请求体 5MB 上限 | 超限返回 413，防 OOM |
| L4 | 日志轮转 | shim.log > 10MB 自动轮转 |
| L5 | PID 文件 | stop-shim 按 PID 精确杀进程树 |

### 4.2 settings.json watcher

- **BASE_URL 自愈**：检测到外部改写 → 热切换上游 → 原子写回 :8788
- **alias map 刷新**：每 tick 全量重建，不持久化
- **trailing comma 容忍**：失败时正则清理后重试

### 4.3 上游超时机制

- 默认 **300 秒（5 分钟）**，从请求发出到首字节到达
- 首字节到达后 `clearTimeout`——SSE 流传输中不再受超时限制
- 可通过 `SHIM_UPSTREAM_TIMEOUT` 自定义

### 4.4 静态模型改写规则

`SHIM_MODEL_REWRITE_RULES` 环境变量，用于覆盖 CC 内部硬编码模型名（如 compaction 发出的 `claude-sonnet-4-6`）：

```powershell
$env:SHIM_MODEL_REWRITE_RULES = "claude-sonnet-4-6:claude-sonnet-4-6-cc"
```

### 4.5 错误诊断落盘（error-dumps/）

上游返回 4xx/5xx 时，shim 会把发给上游的完整请求体（改写后）和收到的完整响应体原样落盘到 `error-dumps/<timestamp>_<statusCode>.json`，覆盖非流式和 SSE 流两种响应形态。因为 shim 只在本机 127.0.0.1 运行，落盘内容可能包含完整对话内容，已加入 `.gitignore`，不会被提交。

这是第二代问题（`reasoning_content` 400）定位的关键——纯靠 curl 手工构造请求怎么都复现不出来，直到靠这个功能拿到真实业务流量触发的完整 1584 条消息请求体，才能用二分法在几分钟内锁定根因。详见第10章。

用 `SHIM_ERROR_DUMP=0` 关闭，`SHIM_ERROR_DUMP_DIR` 自定义目录。

## 6. 设计迭代

### 6.1 关键路线

| 阶段 | 问题 | 方案 | 结果 |
| --- | --- | --- | --- |
| 初始 | DMX 400 | 请求侧 thinking 注入 | 修复深层根因 |
| v1 | Node 不能崩溃退出 | watchdog + crash guard | 五层保护 |
| v2 | 模型名带版本后缀 | 前缀匹配 | 兼容升级 |
| v3 | 适配 cc-switch 切换 | settings.json watcher + 自愈 | 零干预跟随 |
| v4 | `_MODEL` vs `_MODEL_NAME` 语义 | 从 settings 读映射，改写 body.model | 正确路由 |
| v5 | cc-switch 固定写 `_MODEL` | watcher 全量刷新 alias map | 不持久化 |
| v6 | thinking 内容被 CC 丢弃 | 响应侧 signature 清零（SSE + JSON） | 双路修复 |
| v7 | SSE signature_delta 事件未清零 | 新增 `signature_delta` delta 类型处理 | thinking 过程完整显示 |
| v8 | CC 内部硬编码模型名 | `SHIM_MODEL_REWRITE_RULES` | 静态改写 |
| v9 | thinking 推理超时被截断 | clearTimeout 机制 | SSE 不受限 |
| v10 | 三类 settings 字段未全覆盖 | `ANTHROPIC_MODEL` / `REASONING_MODEL` 别名 | 全覆盖 |
| v11 | DMX 新增 `reasoning_content` 校验（第二代问题，见第10章） | 剥离结尾 assistant 消息中未完成的 tool_use | 修复续写请求 400 |
| v12 | DMX 无法处理 Anthropic `image` 块（第三代问题，见第11章） | 剥离 image 块并替换为文本占位 | 修复 image_url 反序列化 400 |

### 6.2 关键教训

1. **先确认变量语义再编码**。`_MODEL` / `_MODEL_NAME` 被来回纠正了三轮。

2. **轮询比 fs.watch 可靠**。Windows 外部进程用 tmp+rename 写文件，fs.watch 漏事件。

3. **不持久化别名映射**。过期映射是错误来源，settings.json 是唯一 source of truth。

4. **代码量刻意控制**。670 行 shim.js 中核心逻辑不到 100 行，其余是防御和基础设施。

5. **停止是真正的停止**。stop-shim 必须确保进程彻底退出，不在任何情况下自动复活。

6. **双路修复缺一不可**。请求侧解决下一轮 400，响应侧解决 CC 丢弃 thinking 内容。

7. **上游校验规则会变，且不会通知你**。第一代问题修复上线一个多月后，DMX 悄悄引入了完全不同的 `reasoning_content` 校验（第10章），旧修复对此毫无作用。不能假设修复一次就一劳永逸——需要错误诊断能力（见 4.5）持续跟踪。

8. **孤立复现前先看真实流量**。第二代问题最初用 curl 手工构造的孤立请求怎么都复现不出来（试了十几种组合都是 200），直到给 shim 加上错误落盘、拿到真实业务流量触发的完整请求体，用二分法在真实数据上定位，才在几分钟内锁定根因。合成测试适合验证假设，但根因定位应该始终以真实抓包数据为准。

9. **DMX 网关存在间歇性非确定性行为**。同一份请求体重放，偶尔会出现结果不一致（如连续 10 次里 1 次结果相反）。定位规律时必须对同一输入重复测试 5-10 次确认稳定性，单次测试结果不可信。

10. **模型变体后缀意味着不同的校验路径**。同一个含 image 块的请求，`deepseek-v4-pro-guan-cc` 能通过，`deepseek-v4-pro-cc` 就 400——说明 DMX 用了不同的请求校验和转换逻辑。出问题时先换模型变体试试，可能是最快的临时规避方案。

11. **不要假设 Anthropic 原生类型对第三方模型安全**。`image` 是标准的 Anthropic content block 类型，CC 用它是完全合理的——但 DeepSeek 不支持，DMX 也没做兼容处理。shim 需要充当"格式净化器"角色，不仅处理 thinking 相关的字段，也要处理所有可能不被上游支持的 Anthropic 原生内容类型。

## 10. 第二代问题：`reasoning_content` 400（2026-07-16）

### 10.1 问题

第一代修复上线一个多月后，DMX 网关悄悄引入了一条新的、完全不同的校验规则，报错变为：

```text
API Error: 400 status_code=400, The reasoning_content in the thinking mode must be passed back to the API.
```

注意字段名是 `` `reasoning_content` ``（DeepSeek 原生 OpenAI 格式的推理字段），不是第一代的 `content[].thinking`。旧的"注入空 thinking 占位块"修复对此完全无效——shim 仍然正常运行、仍然注入了占位块，但请求依然被拒绝。

### 10.2 排查过程

1. **确认 shim 仍在工作**：对照报错请求的 request id 时间戳（`20260715134214...`）与 `shim.log`，发现同一时刻 shim 确实处理了请求并记录了 `FIXED: injected N thinking block(s)`——说明 shim 没有失效，是新校验规则本身让旧修复失效了。

2. **孤立 curl 测试全部失败复现**：手工构造了十几种组合（assistant 缺 thinking + tool_use、空 thinking 占位块、多轮连续无 thinking、纯文本多轮等），全部返回 200 OK。说明这不是一个能靠孤立请求轻易复现的简单场景。

3. **加错误诊断落盘（关键转折点）**：给 shim 新增 `dumpErrorArtifact()`，在上游返回 4xx/5xx 时把完整的请求体（改写后）和响应体存到 `error-dumps/`（详见 4.5）。几分钟后，用户在另一个真实编程任务里再次触发 400，shim 自动捕获了 3 份现场——一份是含 1584 条消息的完整对话历史，重放它对 dmxapi.cn 稳定 100% 复现 400。

4. **二分法在真实数据上定位**：对 1584 条消息的请求体做二分截断（每次都截到一个语义完整的位置），逐步缩小范围：
   - 238 条 → 200，790 条 → 400 → 二分到 466 条 → 200，467 条 → 400。
   - 确认 466/467 两个文件除最后一条消息外逐字节相同（`json.dumps(..., sort_keys=True)` 比对）。
   - 第 467 条（index 466）是一个 `assistant` 消息：`[thinking(占位符), text, tool_use(ExitPlanMode)]`，且是整个请求消息数组的**最后一条**。

5. **排除了多个错误假设**（每一步都用真实数据 + 隔离测试验证，而非拍脑袋）：
   - ❌ 不是"计数上限"——在合法位置插入第 205 个占位 thinking 块，仍然 200。
   - ❌ 不是 `context_management`（`clear_thinking_20251015`）字段——去掉该字段，完整请求依然 400。
   - ❌ 不是总 token 数阈值——把 token 数从 279016 一路填充到 300618+，依然 200。
   - ❌ 不是 `tool_result` 内容长度——同等长度的合成填充文本可以正常通过，前半/后半分别截取也都正常，只有真实完整内容才复现。
   - ❌ 不是 `ExitPlanMode` 这个工具名本身——换成 `Bash` 依然 400。
   - ❌ 不是内容本身的具体字符——用极简 placeholder 消息替换掉真实的 index466 内容，依然 400。

6. **真正的判别实验**：用最小化的两条消息直接测试——

   ```json
   {"role":"user","content":"用Bash echo hello"},
   {"role":"assistant","content":[
     {"type":"thinking","thinking":"","signature":"deepseek-think-fix"},
     {"type":"text","text":"我来执行"},
     {"type":"tool_use","id":"t1","name":"Bash","input":{"command":"echo hello"}}
   ]}
   ```

   请求以这条 assistant 消息**结尾**（没有后续的 user/tool_result），稳定 400。而把最后一条消息换成不含 tool_use 的纯文本续写（`{"type":"text","text":"从前有一个"}`），稳定 200。

### 10.3 根因

**当请求消息数组的最后一条消息是 `role: assistant` 且其 `content` 含 `tool_use` 块时，DMX 稳定返回 `reasoning_content` 400 错误——无论 thinking 块是否存在、是否为空、signature 是什么，都无法规避。**

这类"以未完成 tool_use 结尾"的请求对应 Claude Code 的**续写（continuation）机制**：当 assistant 回复因 `max_tokens` 被截断，且截断点恰好落在一个工具调用的 JSON 参数生成过程中时，CC 会把这条不完整的 assistant 消息作为消息列表最后一条重新发出，要求模型接着把参数补完，而不追加任何新的 user 消息。DMX 现在要求这类"续写"请求必须携带 DeepSeek 原生的 `reasoning_content` 字段来延续推理链——但 Anthropic 消息格式里根本没有这个字段、也无法从 `thinking` 块伪造等价物，无论 shim 怎么填充 thinking 块内容都无法满足这条校验。

**判别条件小结**（均通过反复重复测试 5-10 次确认稳定，见 6.2 教训 9）：

| 场景 | 结果 |
| --- | --- |
| 结尾 assistant 含 tool_use（无论前面是否已有完成的 tool_use、是否有 text、thinking 是否为空/非空/带真实内容） | **400（稳定）** |
| 结尾 assistant 不含 tool_use（纯 text 续写，即使也用了空占位 thinking） | 200（稳定） |
| 结尾是 user 消息（tool_result 或正常提问），历史消息中间有 assistant 缺 thinking | 200（DMX 对这类历史消息的校验已经放宽，第一代问题的原始触发条件在当前 DMX 版本下已不再复现） |

另外发现 DMX 网关本身存在**间歇性非确定性行为**：同一份请求体连续重放，个别情况下会出现结果不一致（例如连续 10 次里 1 次结果相反），怀疑是网关多实例部署下部分实例的校验逻辑尚未完全同步。这是次要噪音，不影响上述主规律的稳定性（10 次里 9-10 次都符合规律）。

### 10.4 修复方案

无法伪造 `reasoning_content`，因此选择降级策略：检测到这种"结尾 assistant 含 tool_use"的请求时，**把该消息里所有 tool_use 块整体移除**（保留 thinking/text），让 DMX 把这次请求当作一次全新的生成请求处理，模型会重新决定要不要调用工具、调用什么工具。

```javascript
// shim.js
function fixTrailingAssistantToolUse(body) {
  if (!body || !Array.isArray(body.messages) || !body.messages.length) return false;
  const last = body.messages[body.messages.length - 1];
  if (!last || last.role !== 'assistant' || !Array.isArray(last.content)) return false;
  if (!last.content.some(b => b && b.type === 'tool_use')) return false;
  last.content = last.content.filter(b => b && b.type !== 'tool_use');
  if (last.content.length === 0) last.content.push(Object.assign({}, PLACEHOLDER_THINKING));
  return true;
}
```

代价：那次被截断的"半成品"工具调用会丢失（例如一个正在生成中、JSON 参数还没写完的工具调用）。但相比于直接 400 硬报错让整个任务卡死、用户必须手动重试，这是可接受的降级——模型会立刻重新生成，通常能重新决定调用同一个工具。

验证：用之前稳定复现 400 的真实请求体（`truncated_467.json`，466 条历史 + 结尾含 tool_use 的 assistant 消息）直接打 shim，日志显示 `FIXED: injected 0 thinking block(s) + dropped trailing unfinished tool_use`，响应变为 200。

### 10.5 与第一代修复的关系

两代问题在 shim 中是**并行的两条独立修复逻辑**，互不影响：

- `fixThinkingRoundtrip()`：遍历所有消息，给"含 tool_use 但完全没有 thinking 块"的 assistant 消息注入占位符（第一代，修复的是"CC 丢弃 thinking 导致下一轮完全没有 thinking"）。
- `fixTrailingAssistantToolUse()`：只检查**最后一条**消息，把"结尾且含未完成 tool_use"的 assistant 消息剥离 tool_use（第二代，修复的是"CC 续写请求携带未完成 tool_use"）。

一次请求可能同时命中两条逻辑（先注入占位 thinking，再剥离结尾 tool_use），shim.log 会记录合并后的动作，如：

```text
FIXED: injected 1 thinking block(s) + dropped trailing unfinished tool_use [model=deepseek-v4-pro-cc]
```

## 11. 第三代问题：`image_url` 解析错误（2026-07-16）

### 11.1 问题

第二代修复提交后不到几小时，用户在使用 `deepseek-v4-pro-cc` 模型时连续三次报错：

```text
API Error: 400 status_code=400, Failed to deserialize the JSON body into the target type:
messages[145]: unknown variant image_url, expected text at line 1 column 407956
```

换了 `deepseek-v4-pro-guan-cc` 模型后恢复正常。用户确认没有发送图片，全是文本操作。

### 11.2 排查过程

1. **错误诊断落盘直接定位**：`error-dumps/` 目录下已捕获到三份 400 现场（16:23、16:25、16:26 各一份），每份都稳定可复现，无需长时间二分。重放请求体到 `dmxapi.cn` 稳定 400。

2. **搜索请求体中的 `image_url`**：`re.finditer('image_url', raw_body)` 只找到三处——全在响应体的错误消息文本中，请求体本身没有任何 `image_url` 内容块。

3. **改搜 `"source"` 字段**（Anthropic image 块的特征字段）：`re.finditer('"source"', raw_body)` 找到一处——在 `messages[126]` 位置，内容是一个 user 消息的工具调用结果，结构为 `[tool_result, text, image, text, text, text, text, text, text]`，其中 `image` 块为 Anthropic 原生格式：

   ```json
   {"type": "image", "source": {"type": "base64", "media_type": "image/png", "data": "iVBORw0KGgo..."}}
   ```

   base64 data 长度约 138KB，是一个完整的 PNG 截图。CC 在工具调用结果中自动混入了截图内容。

4. **确认根因**：DMX 内部的请求格式转换层（Anthropic → OpenAI/DeepSeek）使用严格的 serde 枚举 parser，遇到 `type: image` 的 `source` 子对象时，无法将其映射到 OpenAI 的 `image_url` 类型，直接报反序列化错误。因为 DeepSeek 不支持多模态输入，`image` 类型对 DeepSeek 后端无意义。错误报告中 `messages[145]` 与实际 `messages[126]` 相差 19 个索引，是 DMX 内部消息编号偏移导致的（可能与 system 消息的特殊处理有关），不影响根因判断。

5. **跨模型变体差异**：同一个请求用 `deepseek-v4-pro-guan-cc` 不触发此错误，用 `deepseek-v4-pro-cc` 就报 400。说明 DMX 对这两个模型变体使用了不同的请求校验路径（`-cc` 后缀的变体可能有更严格的格式校验），换模型变体是临时规避方案。

### 11.3 修复方案

修复逻辑很简单——深度遍历所有消息，将 `type: image` 的内容块整体替换为一行紧凑的文本占位，同时保留原块的 `media_type` 和数据大小信息：

```javascript
function fixImageBlocks(body) {
  if (!body || !Array.isArray(body.messages)) return 0;
  let stripped = 0;
  for (const msg of body.messages) {
    if (!msg || !Array.isArray(msg.content)) continue;
    const keep = [];
    for (const block of msg.content) {
      if (!block || block.type !== 'image') { keep.push(block); continue; }
      const mt = (block.source && block.source.media_type) || 'image';
      const dims = [];
      if (block.source && block.source.type === 'base64' && block.source.data) {
        dims.push('base64');
        dims.push(`${Math.round(block.source.data.length / 1024)}KiB`);
      }
      const note = dims.length ? `[image: ${mt}, ${dims.join(', ')}]`
                               : `[image: ${mt}]`;
      keep.push({ type: 'text', text: note });
      stripped++;
    }
    msg.content = keep;
  }
  return stripped;
}
```

替换后的文本占位示例：`[image: image/png, base64, 135KiB]`

代价：截图内容丢失，CC 看不到原始图像。但 CC 仍然知道曾经有图片存在（占位文本），可以在需要时请求用户重新上传。对比直接 400 卡死整个任务，这是可接受的降级。

### 11.4 与修复一、修复二的关系

三者是**并行的三条独立修复逻辑**，各自处理完全不同的触发条件：

- `fixThinkingRoundtrip()` — 遍历所有消息，注入缺失的 thinking 块（第一代）
- `fixTrailingAssistantToolUse()` — 只检查最后一条消息（第二代）
- `fixImageBlocks()` — 遍历所有消息，剥离 image 块（第三代）

一次请求可能同时命中多条逻辑，shim.log 会记录合并动作，如：

```text
FIXED: injected 121 thinking block(s) + stripped 1 image(s) [model=deepseek-v4-pro-guan-cc]
```

## 12. 已知限制

1. **CC 重启问题**：shim 重启后 CC 需重启才能通过 shim 路由。
2. **cc-switch 固定写 `_MODEL` 值**：watcher 检测后立即重建映射，反映当前配置。
3. **`url.parse` 弃用警告**：Node 25 DEP0169，无害。
4. **CC 内部硬编码模型**：compaction 等操作用 `claude-sonnet-4-6`，需通过 `SHIM_MODEL_REWRITE_RULES` 静态改写。
5. **续写请求会丢失半成品工具调用**：第二代修复剥离结尾未完成的 tool_use 后，那次工具调用的部分生成内容丢失，模型需要重新生成（见 10.4）。
6. **DMX 网关校验规则可能再次变化**：两代问题相隔一个多月，说明上游会持续调整校验逻辑，且不会通知调用方。已有 `error-dumps/` 诊断能力（见 4.5），下次出现新的 400 变体时应优先用真实抓包数据排查，而非孤立 curl 猜测。
7. **DMX 网关存在间歇性非确定性行为**：同一请求体重放可能小概率出现结果不一致（见 10.3），确认任何新规律都应重复测试 5-10 次。
8. **截图/图像内容会丢失**：第三代修复将 image 块替换为文本占位 `[image: ...]`，原始图像数据丢失。CC 可通过占位文本知道图像存在，在需要时请求用户重新上传。
9. **不同 deepseek 模型变体校验路径不同**：`-cc` 后缀的变体可能使用更严格的请求校验（见 11.2 排查过程 #5），同一请求在 `-gcc` 变体上正常但在 `-cc` 上 400 是可能的。临时规避可换变体，永久修复依赖 shim 的内容净化。
