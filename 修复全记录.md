# DeepSeek Think 模式修复 — 完整记录

> 从报错到根因定位，到可维护、可适配多模式、零干预的最终方案。  
> 最后一次更新：2026-06-01

---

## 1. 问题

Claude Code 通过 API 聚合商（DMX 等）调用 DeepSeek 等第三方模型时，thinking 模式下报错：

```
API Error: 400 The content[].thinking in the thinking mode must be passed back to the API.
```

触发条件：EFFORT Extra high（启用 extended thinking）+ multi-turn + tool_use。

---

## 2. 根因

### 2.1 直接原因（两个机制叠加）

1. **CC 丢弃 thinking 块**：DeepSeek 经 DMX 后返回的 `thinking` 块 `signature` 是伪 UUID，不符合 Anthropic 签名规范，CC 在构建下一轮消息时丢弃它。
2. **上游 round-trip 校验**：DMX 要求如果 assistant 消息含 `tool_use`，则必须同时有 `thinking` 块。

两步叠加：CC 丢弃 thinking → 下一轮 assistant 只剩 `tool_use` → DMX 400。

### 2.2 关键实验

通过阶梯式 curl 探测确认根因：

| 实验 | assistant 消息内容 | 结果 |
| --- | --- | --- |
| B1 | `[thinking(UUID签名), tool_use]` | 200 |
| B2 | `[tool_use]`（无 thinking） | **400** — 精确复现 |
| B3 | `[thinking(空签名""), tool_use]` | 200 |
| B4 | `[thinking(签名"x"), tool_use]` | 200 |

**决定性证据**：上游只校验 thinking 块的存在性，不校验内容。

### 2.3 因果链

```
DeepSeek 无条件返回 thinking 块
  → CC 检测 signature 不是真 Anthropic 签名，丢弃 thinking 块
  → 下一轮 assistant 只剩 tool_use
  → DMX 400：content[].thinking must be passed back
```

---

## 3. 修复方案

### 3.1 核心思路

在 CC 和上游之间插入本地反向代理（shim），双路修复：

**请求侧**：检测到 deepseek 请求时，在 assistant 消息前注入占位 thinking 块。
```json
{ "type": "thinking", "thinking": "", "signature": "deepseek-think-fix" }
```

**响应侧**：在上游返回的 SSE/JSON 中，将非标准 signature 清空，让 CC 保留 thinking 块内容不丢弃。

### 3.2 当前架构

```
CC ──▶ shim :8788 (shim.js) ──▶ upstream (cc-switch 代理 或 dmxapi.cn 直连)
          │
          ├─ 请求侧处理（POST /v1/messages）：
          │    ├─ 读 settings.json 构建三类别名映射
          │    ├─ 查静态改写规则（SHIM_MODEL_REWRITE_RULES）
          │    ├─ 改写 body.model 从 CC 标签到真实模型名
          │    └─ deepseek-* 且缺 thinking 块 → 注入占位块
          ├─ 响应侧处理：
          │    ├─ 非流式 JSON：thinking signature 清零
          │    └─ SSE 流：逐 event 改写 thinking signature
          ├─ 每 3 秒轮询 settings.json：
          │    ├─ BASE_URL 自愈 + 别名映射刷新
          │    └─ trailing comma 容忍性解析
          ├─ GET /health → 返回运行状态 JSON
          └─ 崩溃 → watchdog 3 秒后重启
```

### 3.3 模型识别（三类字段全覆盖）

shim 从 `settings.json` 中读取三类模型字段，构建别名映射：

| 字段类型 | shim 用途 |
| --- | --- |
| `ANTHROPIC_DEFAULT_<SLOT>_MODEL` / `_MODEL_NAME` | 映射 key → 真实模型名 |
| `ANTHROPIC_REASONING_MODEL` / `_MODEL_NAME` | 同上 |
| `ANTHROPIC_MODEL` / `_MODEL_NAME` | 同上 |

**解析优先级**：`SHIM_MODEL_REWRITE_RULES` > settings.json 别名映射 > identity（透传）

## 4. 进程管理与自愈

### 4.1 进程保护

| 层 | 机制 | 说明 |
| --- | --- | --- |
| L1 | watchdog 循环守护 | Node 崩溃 → 3 秒后自动重启 |
| L2 | uncaughtException / unhandledRejection | 致命异常记日志后 exit，触发 L1 |
| L3 | 请求体 5MB 上限 | 超限返回 413，防 OOM |
| L4 | 日志轮转 | shim.log > 10MB 自动轮转 |
| L5 | PID 文件 | stop-shim 按 PID 精确杀进程树 |

### 4.2 settings.json watcher

- **BASE_URL 自愈**：检测到外部改写 → 热切换上游 → 原子写回 :8788
- **alias map 刷新**：每 tick 全量重建，不持久化
- **trailing comma 容忍**：失败时正则清理后重试

### 4.3 上游超时机制

- 默认 **300 秒（5 分钟）**，从请求发出到首字节到达
- 首字节到达后 `clearTimeout`——SSE 流传输中不再受超时限制
- 可通过 `SHIM_UPSTREAM_TIMEOUT` 自定义

### 4.4 静态模型改写规则

`SHIM_MODEL_REWRITE_RULES` 环境变量，用于覆盖 CC 内部硬编码模型名（如 compaction 发出的 `claude-sonnet-4-6`）：

```powershell
$env:SHIM_MODEL_REWRITE_RULES = "claude-sonnet-4-6:claude-sonnet-4-6-cc"
```

## 6. 设计迭代

### 6.1 关键路线

| 阶段 | 问题 | 方案 | 结果 |
| --- | --- | --- | --- |
| 初始 | DMX 400 | 请求侧 thinking 注入 | 修复深层根因 |
| v1 | Node 不能崩溃退出 | watchdog + crash guard | 五层保护 |
| v2 | 模型名带版本后缀 | 前缀匹配 | 兼容升级 |
| v3 | 适配 cc-switch 切换 | settings.json watcher + 自愈 | 零干预跟随 |
| v4 | `_MODEL` vs `_MODEL_NAME` 语义 | 从 settings 读映射，改写 body.model | 正确路由 |
| v5 | cc-switch 固定写 `_MODEL` | watcher 全量刷新 alias map | 不持久化 |
| v6 | thinking 内容被 CC 丢弃 | 响应侧 signature 清零（SSE + JSON） | 双路修复 |
| v7 | CC 内部硬编码模型名 | `SHIM_MODEL_REWRITE_RULES` | 静态改写 |
| v8 | thinking 推理超时被截断 | clearTimeout 机制 | SSE 不受限 |
| v9 | 三类 settings 字段未全覆盖 | `ANTHROPIC_MODEL` / `REASONING_MODEL` 别名 | 全覆盖 |

### 6.2 关键教训

1. **先确认变量语义再编码**。`_MODEL` / `_MODEL_NAME` 被来回纠正了三轮。

2. **轮询比 fs.watch 可靠**。Windows 外部进程用 tmp+rename 写文件，fs.watch 漏事件。

3. **不持久化别名映射**。过期映射是错误来源，settings.json 是唯一 source of truth。

4. **代码量刻意控制**。670 行 shim.js 中核心逻辑不到 100 行，其余是防御和基础设施。

5. **停止是真正的停止**。stop-shim 必须确保进程彻底退出，不在任何情况下自动复活。

6. **双路修复缺一不可**。请求侧解决下一轮 400，响应侧解决 CC 丢弃 thinking 内容。

## 9. 已知限制

1. **CC 重启问题**：shim 重启后 CC 需重启才能通过 shim 路由。
2. **cc-switch 固定写 `_MODEL` 值**：watcher 检测后立即重建映射，反映当前配置。
3. **`url.parse` 弃用警告**：Node 25 DEP0169，无害。
4. **CC 内部硬编码模型**：compaction 等操作用 `claude-sonnet-4-6`，需通过 `SHIM_MODEL_REWRITE_RULES` 静态改写。
