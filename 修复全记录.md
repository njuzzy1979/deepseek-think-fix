# DeepSeek Think 模式修复 — 完整记录

> 从报错到根因定位，到可维护、可适配多模式、零干预的最终方案。  
> 最后一次更新：2026-05-28

---

## 1. 问题

Claude Code 通过 DMX（dmxapi.cn）调用 deepseek 模型时，thinking 模式下报错：

```
API Error: 400 The content[].thinking in the thinking mode must be passed back to the API.
```

触发条件：EFFORT Extra high（启用 extended thinking）+ multi-turn + tool_use。

---

## 2. 根因

### 2.1 直接原因

DeepSeek 经 DMX 后**无条件**返回 thinking 块。DMX 的 round-trip 校验要求：如果 assistant 消息含 `tool_use`，则必须同时有 `thinking` 块。

Claude Code 收到 thinking 块后，其 `signature` 是伪 UUID（DMX 用 message_id 充当），不符合 Anthropic 签名预期，**CC 丢弃了 thinking 块**。下一轮请求时 assistant 只剩 `tool_use`，DMX 校验失败。

### 2.2 关键实验

通过阶梯式 curl 探测确认根因：

| 实验 | assistant 消息内容 | 结果 |
|------|-------------------|------|
| B1 | `[thinking(UUID签名), tool_use]` | 200 |
| B2 | `[tool_use]`（无 thinking） | **400** — 精确复现 |
| B3 | `[thinking(空签名""), tool_use]` | 200 |
| B4 | `[thinking(签名"x"), tool_use]` | 200 |

**决定性证据**：B3/B4 签名完全伪造，上游仍然接受。上游**只校验 thinking 块的存在性，不校验内容**。

### 2.3 因果链

```
DeepSeek 无条件返回 thinking 块
  → CC 检测 signature 不是真 Anthropic 签名，丢弃 thinking 块
  → 下一轮 assistant 只剩 tool_use
  → DMX 400：content[].thinking must be passed back
```

---

## 3. 修复方案

### 3.1 核心思路

在 CC 和上游之间插入本地反向代理（shim），检测到 deepseek 请求时在请求体的 assistant 消息前插入占位 thinking 块。

占位块内容：
```json
{ "type": "thinking", "thinking": "", "signature": "deepseek-think-fix" }
```

### 3.2 架构

```
CC ──▶ shim :8788 ──▶ upstream (cc-switch 或 dmxapi.cn)
          │
          ├─ 读 settings.json → _MODEL → _MODEL_NAME 映射
          ├─ 改写 body.model → 真实模型名
          ├─ deepseek-* + 缺 thinking → 注入占位块
          ├─ 其他模型 → 原样透传
          ├─ 每 3s 轮询 settings.json：BASE_URL 自愈 + 映射刷新
          └─ crash → watchdog 3s 重启
```

### 3.3 模型识别

shim 从 `~/.claude/settings.json` 中读取模型信息。语义：

| 字段 | 含义 | shim 用做什么 |
|------|------|-------------|
| `_MODEL` | 显示标签，CC 发到 body.model 的值 | 映射 key |
| `_MODEL_NAME` | **实际调用的模型** | 映射目标 |

识别流程：
1. CC 发请求 → body.model = `_MODEL` 值
2. shim 查映射表 → 真实模型 = `_MODEL_NAME` 值 → 改写 body.model
3. 真实模型以 `deepseek` 开头 → 注入 thinking 块
4. 真实模型不以 `deepseek` 开头 → 原样透传

该方案兼容两种 cc-switch 模式：
- **代理模式**：cc-switch 在 shim 下游，改写后 cc-switch 按模型名路由
- **直连模式**：上游直接就是 dmxapi.cn，改写确保上游调用正确的模型

## 4. 进程管理与自愈

### 4.1 崩溃自恢复（watchdog）

start-shim.ps1 生成 `.watchdog.cmd` 并在后台启动。watchdog 循环运行 node shim.js；如果 node 崩溃（OOM、未捕获异常），等 3 秒后重启。shim.js 本身有 `uncaughtException` / `unhandledRejection` 兜底，记录日志后 exit 触发重启。

停止机制：stop-shim.ps1 先删除 `.shim-pid.txt`，watchdog 检测到 PID 文件消失后自行退出，不再重启 node。然后杀整个进程树，还原 settings.json。

### 4.2 settings.json watcher

cc-switch 托盘程序持续重写 settings.json（接管 `ANTHROPIC_BASE_URL` 及其他字段）。shim 内置 3 秒轮询 watcher：

- **BASE_URL 自愈**：检测到被改成非 :8788 地址 → 将新值设为 shim 上游（热切换 HTTP/HTTPS）→ 原子写回 :8788 → 记录到 last-upstream.txt
- **alias map 刷新**：每 tick 全量重建 `_MODEL → _MODEL_NAME` 映射，以 settings.json 当前状态为准，不持久化到磁盘

### 4.3 生命周期

- **启动**：仅手动双击 start-shim.cmd（或 ps1）。不注册任何开机自启。
- **停止**：双击 stop-shim.cmd。彻底停止 watchdog + node。不会自动重启，直到下次手动 start。
- **崩溃**：watchdog 在 3 秒后重启 node，日志记录 `shim crashed ... restarting in 3s`。

## 6. 设计迭代

### 6.1 关键路线

| 阶段 | 问题 | 方案 | 结果 |
|------|------|------|------|
| 初始 | DMX 400 | request-side thinking 注入 | 修复深层根因 |
| v1 | Node 不能崩溃退出 | watchdog + crash guard | 四层保护 |
| v2 | 模型名带版本后缀（claude-haiku-4-5-20251001）| 前缀匹配 | 兼容 CC 版本升级 |
| v3 | 需要适配 cc-switch 切换 provider | settings.json watcher + BASE_URL 自愈 | 零干预跟随 |
| v4 | `_MODEL` vs `_MODEL_NAME` 语义澄清 | 从 settings 读映射，改写 body.model | 保证上游收到正确模型 |
| v5 | cc-switch 代理模式下 `_MODEL` 被它固定写回 | watcher 全量刷新 alias map | 不持久化，settings.json 永远为源 |

### 6.2 关键教训

1. **先确认变量语义再设计代码**。`_MODEL` / `_MODEL_NAME` 的定义被来回纠正了三轮，每次都导致大段代码的增删。如果用最小化端到端实例验证语义，再编码，会省很多时间。

2. **轮询比 fs.watch 可靠**。Windows 上外部进程（cc-switch、PowerShell）写 settings.json 通常用 tmp+rename 模式，fs.watch 经常丢事件。3 秒轮询慢一点但稳定，且对用户体验无感知（CC 每次 API 请求间隔本身就 >1s）。

3. **"只增不减"的记忆策略不适用于可变的配置映射**。早期尝试把 alias map 持久化到磁盘做 warm-start + 累积，但用户切换配置后旧映射变成了错误来源。当前方案不持久化，watcher 每 tick 全量重建，settings.json 是唯一 source of truth。

4. **更少的代码 = 更少的维护和更强的可理解性**。410 行 shim.js 中，核心逻辑（匹配 + 注入 + 改写 + 透传）不到 60 行，剩余是 watcher、watchdog、日志、错误处理。代码量被刻意控制，方便审查和修改。

5. **停止必须是真停止，不能"随时会重启"**。stop-shim.ps1 必须在 stop 时确保进程彻底退出，不在任何情况下自动复活——用户双击 stop 就得停，双击 start 才活。watchdog 仅处理 crash，不处理 stop。

## 9. 已知限制

1. **shim 重启时，CC 尚未重启** —— CC 的内存 alias 可能还指向旧值，但 shim 已从 settings.json 全量重建映射。如果用户在中间改过配置，CC 内存和 settings.json 不同步，shim 按 settings.json 当前值行事。这是 CC 的设计行为（启动时读一次 env），不是 shim 能控制的。

2. **cc-switch 固定写 `_MODEL` 值** —— shim 的 watcher 检测到 cc-switch 重写 settings.json 后会立即重建 alias map（从最新的 settings.json 读取），所以映射始终反映 cc-switch 当前的实际配置。

3. **`shim.js` 使用 `url.parse`** —— Node 25 会产生 DEP0169 警告，不影响功能，不计划升级到 WHATWG URL API（避免增加代码复杂度）。
